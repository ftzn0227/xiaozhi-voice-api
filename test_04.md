## 6. 遇到的问题与解决方案

### 6.1 会话记忆丢失
**问题**：第二问引用第一问的名字时失败。  
**原因**：DeepSeek 模型调用时没有将 `history` 传入 `messages`。  
**解决**：在 `DeepSeekQA.ask` 中增加 `history` 参数，拼接在系统提示词后、当前问题前。同时 `ConversationMemory` 需正确 append 用户和助手消息。

### 6.2 意图分类 JSON 解析失败
**问题**：分类器返回的字符串不是合法 JSON，导致服务 500。  
**原因**：DeepSeek 输出有时会附带解释文字。  
**解决**：  
- 在调用时设置 `response_format={"type": "json_object"}`（API 支持时）。  
- 增加异常捕获，若解析失败则降级为 `simple_qa`，保证服务可用。

### 6.3 不同 worker 下会话不共享
**问题**：使用 `gunicorn` 多 worker 时，内存字典无法跨进程共享。  
**原因**：当前 `ConversationMemory` 使用进程内字典。  
**解决**：临时方案使用单 worker 测试；生产环境需要替换为 Redis 或数据库。

### 6.4 推荐问题与上下文无关
**问题**：推荐问题泛泛，如“你好吗”之类。  
**原因**：提示词中未注入对话历史，模型无法关联上下文。  
**解决**：在 `get_suggestions` 中加入最近一轮的用户消息和助手回答作为上下文。

## 7. 各模块优化方案

### 7.1 对话记忆模块
- **当前**：内存存储，易丢失，不支持多 worker。  
- **优化**：引入 Redis，使用 `session_id` 作为 key，list 存储历史，TTL 自动过期。  
- **高级**：加入记忆摘要功能，长对话超出 token 限制时，用 LLM 对早期历史生成摘要。

### 7.2 意图路由模块
- **当前**：每次请求都调用一次 LLM 分类，增加延迟和成本。  
- **优化**：  
  - 增加关键词规则预筛（如“计算”“天气”直接判为 multi_step，短问候直接判为 simple_qa），仅模糊问题才调用 LLM。  
  - 缓存相同问题的分类结果（短时间内的重复问题）。  
- **扩展**：真实工具调用（Function Calling），让模型输出 `tool_calls`，系统执行后返回结果继续推理。

### 7.3 推荐系统
- **当前**：依赖一次额外的 LLM 请求。  
- **优化**：  
  - 与主回答共用一次 LLM 调用，在 system prompt 中要求模型同时输出 `suggestions`，节省一次请求。  
  - 缓存高频问题的推荐列表。

## 8. 心得体会
通过阶段四的开发与测试，深刻体会到上下文管理是让 AI 从“单轮应答”迈向“连续助手”的关键。简单的内存字典配合历史注入即可快速实现多轮记忆，但实际生产必须考虑持久化与并发。

意图路由的设计初看简单，但要让模型稳定输出 JSON 并不容易，需要调优提示词并做好降级处理。这种“意图-分治”模式为后续接入搜索、数据库等工具打下了清晰的结构。

推荐系统虽然可选，但能明显提升用户体验，让对话更自然、更富探索性。而保持原有接口兼容性的同时扩展新功能，让我学会了如何通过参数扩展（如 `session_id`）和平滑降级来保证代码的稳健性。

整个项目从硬件耦合的 ESP32 服务端，到解耦为通用 FastAPI 服务，再到增添智能对话能力，每一步都加深了对异步编程、API 设计和模块化架构的理解。
